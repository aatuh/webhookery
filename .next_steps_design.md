# Webhookery Next Steps Design

Date: 2026-05-27

Status: design proposal for the next product phase

Source inputs:

- `.GitHub-Project-Recommendations-2026-05-27.html`
- `.Marketing-For-Profit-GitHub-Projects-2026-05-27.html`
- `.initial_design.md`
- current repository implementation, docs, release, and operations artifacts

This document describes the next phase for Webhookery after the core
production-respectable self-hosted product work. It is intentionally not a
generic feature backlog. The next phase is productization: make the existing
webhook evidence system easier to evaluate, trust, self-host, compare, buy, and
operate without weakening its core claims.

## Executive Summary

Webhookery should be productized first among the related project ideas. The
reason is practical: the pain is narrow, common, easy to explain, and directly
matched by the current implementation.

The next design phase should package Webhookery as:

> Self-hosted webhook evidence infrastructure for teams that need durable
> capture, replay, auditability, and procurement-friendly commercial rights.

The product should not try to compete as a generic webhook sender. Its strongest
wedge is evidence: proof of receipt, verification, routing decisions, delivery
attempts, replay actions, retention actions, reconciliation results, audit-chain
continuity, and release evidence.

The next phase therefore focuses on:

1. A public release-candidate product surface.
2. A clean commercial evaluation path.
3. A credible demo and buyer journey.
4. Evidence-backed production readiness material.
5. Honest comparison and education content.
6. A pilot-feedback loop that feeds future engineering priorities.

## Current Product Baseline

The repository is implementation-bearing. It contains Go source, PostgreSQL
migrations, OpenAPI and SDK artifacts, Docker Compose, deployment profiles,
release evidence checks, provider conformance checks, operational docs, and CI
workflows.

The current product direction is consistent with `.initial_design.md`:

- durable capture before success
- raw webhook evidence preservation
- provider-aware verification
- at-least-once delivery
- explicit retries, DLQ, replay, and reconciliation
- SSRF-safe outbound delivery
- tenant-scoped control-plane APIs
- audit evidence and reproducibility
- self-hosted PostgreSQL-first operation

The current production posture should be described as controlled self-hosted
release-candidate maturity, not broad commodity production maturity. The repo
has strong internal evidence, but public adoption still needs release artifacts,
comparison material, demos, clear commercial packaging, and field feedback.

## Source Synthesis

### GitHub Project Recommendation Findings

The GitHub project recommendations identify Webhookery as the first product to
commercialize because it has a coherent wedge:

- Companies depend on webhooks for payments, billing, provisioning, lifecycle
  automation, auth, CRM, marketplaces, and partner integrations.
- Webhook failures are operationally painful because teams often cannot prove
  what happened.
- The highest-value problem is not sending HTTP requests; it is creating a
  durable evidence trail around receipt, verification, routing, delivery,
  replay, and audit.
- Webhookery should be positioned as a system of record for webhook events,
  attempts, signatures, replay decisions, routing changes, retention, DLQ
  recovery, and evidence exports.

The recommended cross-product sequence is:

1. Webhookery proves event delivery and replay.
2. Evidence Ledger, if built later, proves software, release, and security
   posture.
3. Licensing API is built only when needed to monetize Webhookery and related
   products.
4. Agent-aware access control and contract-notary ideas remain later adjacent
   products, not dependencies for Webhookery's next phase.

For Webhookery, the core buyer questions are:

- Did we receive the event?
- Was the provider signature valid?
- What exactly was stored?
- Which route matched?
- Which endpoint was attempted?
- Why did delivery fail?
- Was it retried or replayed?
- Who replayed it and why?
- Can we prove it was not silently lost or modified?

### Marketing Findings

The marketing analysis also recommends Webhookery first. The strongest public
positioning is:

> Webhookery: self-hosted webhook evidence infrastructure for teams that need
> durable capture, replay, auditability, and procurement-friendly commercial
> rights.

The recommended wedge is not "another webhook gateway." Existing products have
validated the category, but Webhookery should differentiate through
self-hosting, evidence, auditability, release reviewability, and commercial
license exceptions.

The recommended near-term commercialization path is:

- publish a first public release, likely `v0.1.0-rc1` or `v0.1.0-alpha`
- add a landing page or landing document
- add a screenshot, diagram, or short demo video/GIF
- create a release evidence package
- publish a production-readiness checklist
- publish technical content around webhook evidence and failure modes
- offer commercial evaluation and production-readiness review packages

The recommended buyer segments are:

- regulated SaaS teams
- fintech and billing platforms
- B2B platforms with partner integrations
- marketplaces
- healthcare-adjacent platforms
- internal platform teams
- teams receiving Stripe, Shopify, GitHub, Slack, and internal product events
- teams with self-hosting, security review, or procurement constraints

## Product Thesis

Webhookery's product thesis should stay narrow:

> Webhookery is the self-hosted control plane for proving what happened to
> webhook events and safely acting on that evidence.

This framing is stronger than:

- "webhook gateway"
- "event bus"
- "workflow automation"
- "integration platform"
- "exactly-once webhook delivery"
- "provider completeness guarantee"

The design should keep the product centered on evidence and recovery:

- capture before acknowledgement
- exact raw evidence preservation
- verification metadata
- normalized evidence envelopes
- deterministic transformation and payload evidence
- signed delivery attempts
- retry and DLQ evidence
- replay evidence
- reconciliation gap evidence
- retention and export evidence
- audit-chain verification
- release evidence for the software itself

## Next Phase Goal

Create a public, self-hosted Webhookery release-candidate package that a
security-conscious team can evaluate in one sitting, run locally, inspect
evidence from end to end, and understand the commercial support path.

The output of this phase should be:

- a tagged release candidate
- release notes with explicit claims and non-claims
- a clean evaluator quickstart
- a public trust/commercial surface
- a reproducible demo
- comparison and educational content
- release evidence artifacts
- a pilot feedback mechanism

## Non-Goals

This phase should not add unrelated enterprise or platform expansion:

- no hosted SaaS offering
- no SAML
- no marketplace plugins
- no arbitrary transformation scripting
- no GraphQL API
- no Kafka, NATS, or Redis authority layer
- no multi-region active-active architecture
- no external timestamping
- no HSM/PKCS#11 adapter
- no compliance certification claim
- no exactly-once delivery claim
- no provider-side completeness guarantee
- no live-provider calls in release acceptance tests
- no vendor-specific SIEM, PagerDuty, Slack, or email notification apps unless a
  paying pilot proves the need

The next phase may reference future adjacent products, but Webhookery must not
depend on them.

## Design Principles

### Evidence First

Every product surface should reinforce that Webhookery is evidence
infrastructure. The public demo, README, landing page, and release notes should
all show the same chain:

1. receive event
2. preserve raw bytes and headers
3. verify provider signature
4. store durable evidence
5. route with explainable config
6. deliver signed payload
7. retry or DLQ on failure
8. replay with reason and audit trail
9. verify audit chain and export evidence

### Honest Claims

Marketing and docs must preserve the current product promise:

- inbound success means durable capture, not downstream success
- delivery is at-least-once
- duplicate events remain visible
- replay creates new work and never mutates original history
- provider reconciliation is limited by provider APIs
- release evidence is not compliance certification

### Self-Hosted Trust

The product should appeal to teams that need control:

- source-visible code
- self-hosted deployment
- PostgreSQL as source of truth
- Docker Compose and Kubernetes paths
- explicit secret boundaries
- commercial license exceptions for non-AGPL use
- release evidence packages
- support and production-readiness review options

### Public Evaluation Before Broad Launch

Do not launch broadly before the public evaluator path is strong. A weak launch
would attract attention before the product has a simple, credible story.

The correct order is:

1. release candidate
2. evaluator quickstart
3. landing/trust surface
4. demo artifact
5. comparison/education content
6. targeted outreach and small launch
7. broad launch only after feedback

### Demo Surface

The next phase needs a demo that makes the evidence story visible.

Recommended demo:

> Fake Stripe-style event to durable capture, signed delivery, failed retry,
> replay, and audit-chain verification.

The demo should show:

- provider-style signed event ingestion
- raw evidence and verification metadata
- route explanation
- delivery payload hash
- signed outbound delivery
- receiver failure and retry
- DLQ or replay path
- audit timeline
- audit-chain verification
- evidence export or bundle verification

Implementation shape:

- use Docker Compose and local fake services only
- no real Stripe/GitHub/Shopify/Slack calls
- no external cloud services
- deterministic sample tenant/source/endpoint
- a script under `examples/` or `scripts/`
- sanitized output suitable for screenshots/video

Acceptance criteria:

- a new evaluator can run it from a clean checkout
- every command has expected output
- failure states are visible, not hidden
- no secrets or private payloads are printed
- demo remains covered by `make rc-check` or an explicit smoke target

### Commercial Trust Surface

The commercial path should be explicit but conservative. The repo already has
governance and commercial boundary files; the next phase should make them
buyer-readable.

Recommended offers:

- Commercial Evaluation
- Release Evidence Package
- Production Readiness Review
- Commercial License + Support
- Custom Integration or Provider Adapter

Potential public price anchors from the marketing input:

- Commercial Evaluation: EUR 490-1,000
- Release Evidence Package: EUR 2,500-5,000
- Production Readiness Review: EUR 7,500-12,500
- Commercial License + Support: EUR 9,900-24,900 per year
- Custom Integration/Provider Adapter: hourly or fixed scope

These should be treated as owner-approved business decisions, not hard-coded
technical requirements. If included publicly, they should be phrased as example
starting ranges or "from" pricing and reviewed before publication.

### Comparison Surface

Comparison content should be honest and useful. It should not attack
competitors or claim superiority without evidence.

Recommended comparison pages:

- Webhookery vs Hookdeck: self-hosted webhook evidence and replay.
- Webhookery vs Svix: inbound evidence control plane vs outbound webhook
  delivery.
- Webhookery vs Convoy: audit-first webhook operations.
- Build vs buy: when to self-host webhook evidence infrastructure.

Comparison principle:

> If a team needs hosted massive scale and low operational ownership, a mature
> hosted vendor may be a better fit. If a team needs self-hosted durable capture,
> audit evidence, private modifications, and commercial licensing, evaluate
> Webhookery.

## Architecture Implications

This phase should not require major runtime architecture changes. Most work is
packaging, documentation, examples, release evidence, and evaluation flow.

If implementation is needed, it should follow existing architecture:

- CLI-first operator commands should use existing application services.
- Demo scripts should call public APIs and CLI commands.
- UI screenshots should reflect authenticated UI behavior, not mocked product
  claims.
- Release evidence should be generated from existing gates, not hand-written.
- Any new smoke scripts must use local fake providers and fake receivers.

Security invariants remain unchanged:

- no 2xx before durable capture
- no route for invalid provider signatures
- no raw payload exposure without elevated permission
- no secret output in CLI, logs, docs, demo artifacts, or release evidence
- no provider completeness guarantee
- no exactly-once claim

## Success Metrics

The next phase succeeds when Webhookery has:

- a public release candidate
- a working self-hosted evaluator path
- a demo that visibly proves the evidence loop
- release evidence artifacts that can be independently inspected
- a clear commercial evaluation path
- honest fit/non-fit messaging
- at least a small number of qualified conversations with target users
- pilot feedback that identifies the next engineering priorities

The phase should not be judged primarily by stars. Stars can help distribution,
but the main signal is whether teams with real webhook incident, replay, audit,
or self-hosting pain ask to evaluate or buy.

Control:

- lead with evidence, audit, replay, and self-hosting
- avoid generic integration-platform language
- show event timelines and audit verification in demos

Control:

- build a deterministic local demo

- keep the story narrow and observable

### Risk: Commercial Terms Are Ambiguous

Control:

- keep AGPL public license clear
- document commercial exception path
- document support boundaries
- owner-review pricing and legal language

### Risk: Comparison Pages Become Inaccurate

Control:

- verify public competitor claims before publishing
- date comparison pages
- focus on buyer fit rather than superiority

### Risk: Support Burden Arrives Before Product Signal

Control:

- use clear support docs
- provide templates for good bug reports
- define no-SLA community expectations
- route serious production users to commercial evaluation

The first public artifact should be a cautious release candidate or alpha, not
a broad GA claim. This keeps trust aligned with current adoption evidence.

### Decision 3 - Evidence Is The Differentiator

The differentiator is not HTTP delivery. It is durable, inspectable evidence
across receipt, verification, routing, delivery, replay, retention,
reconciliation, and audit.

### Decision 4 - Commercial Path Is Explicit

The public AGPL project should clearly explain commercial license exceptions,
support boundaries, release evidence packages, and production-readiness review
options.

### Decision 5 - Adjacent Products Stay Adjacent

Evidence Ledger, Licensing API, agent-aware access control, and contract notary
ideas remain future or separate product lines. They should not distract from
Webhookery's release-candidate productization.
